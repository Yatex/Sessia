# Sessia AI architecture

## Current production flow

WhatsApp webhooks are authenticated by Rails and routed to a tenant-scoped `Client`. The inbound message is persisted before `Ai::InboundMessageProcessor` creates an `AiTask`. `Ai::TaskProcessor` resolves an instruction, calls the separate Vercel AI SDK decision service, executes an allowlisted action in Rails, and stores the outcome in `AiTask.result_data`. Proactive tasks are generated by `sessia:ai:loop`. Client replies and pre-session confirmation requests always use the grounded V2 processor; the remaining proactive workflows continue through their existing deterministic or legacy paths until migrated.

## Grounded inbound flow

Inbound client replies and pre-session confirmation requests always use this sequence:

1. Rails resolves workspace, professional, client, session, message, permissions, locale, and time zone.
2. Rails signs that scope in a short-lived context token.
3. Read-only tools return limited client, session, conversation, settings, availability, payment, and policy data.
4. Tool facts receive stable evidence IDs.
5. The decision service returns a structured candidate with evidence citations.
6. Rails grounds and validates the candidate independently.
7. Only a validated, allowlisted effect reaches `Ai::ActionExecutor`.
8. Candidate, evidence, validation, execution, errors, and latency are persisted in `AiTask.result_data`.

The model never receives database access and cannot mutate payments. Mercado Pago webhooks and explicit professional actions remain the only payment writers.

## Migration boundary

Reminder, no-response follow-up, rescheduling, payment reminder, and post-session feedback tasks still use their existing deterministic or legacy contracts. They should be migrated one trigger at a time after the grounded paths have production evidence.

## Grounded V2

Grounded V2 is the fixed architecture for model-interpreted inbound replies and confirmation requests; it has no rollout ENV flags. V2 sends a minimal signed context to the decision service. The Vercel AI SDK model can call only the allowlisted read-only tools (`client_context`, `session_context`, `conversation_history`, `pending_interaction`, `professional_settings`, `availability_options`, `payment_status`, and `workspace_policies`) through the authenticated Rails endpoint. Rails re-runs exactly the requested tools before validating evidence.

Use the same strong `SESSIA_AI_TOOL_SECRET` in Rails and `ai-decision-service`. Never expose it to a browser or model prompt.

Each task has a stable `idempotency_key`, separate pipeline statuses, one `AiTrace`, and zero or more `MessageDeliveryAttempt` records. Delivery retries reuse the existing message and decision. A template/configuration failure is terminal; temporary provider failures use bounded exponential backoff.

## Verifying Twilio Content templates

Sessia now audits Content resources and WhatsApp approval status through Twilio's official API. The central contracts, production commands, ENV mapping, and manual approval procedure are documented in [whatsapp-templates.md](whatsapp-templates.md). The admin trace continues to show template variables and provider errors without exposing Twilio credentials.
