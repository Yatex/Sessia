# Sessia AI architecture

## Current production flow

WhatsApp webhooks are authenticated by Rails and routed to a tenant-scoped `Client`. The inbound message is persisted before `Ai::InboundMessageProcessor` creates an `AiTask`. `Ai::TaskProcessor` resolves an instruction, calls the separate Vercel AI SDK decision service, executes an allowlisted action in Rails, and stores the outcome in `AiTask.result_data`. Proactive confirmation, reminder, follow-up, feedback, and payment tasks are generated by `sessia:ai:loop` and currently use the legacy processor.

## Grounded inbound flow

When `SESSIA_AI_GROUNDED_INBOUND_ENABLED=true`, inbound client replies use this sequence:

1. Rails resolves workspace, professional, client, session, message, permissions, locale, and time zone.
2. Rails signs that scope in a short-lived context token.
3. Read-only tools return limited client, session, conversation, settings, availability, payment, and policy data.
4. Tool facts receive stable evidence IDs.
5. The decision service returns a structured candidate with evidence citations.
6. Rails grounds and validates the candidate independently.
7. Only a validated, allowlisted effect reaches `Ai::ActionExecutor`.
8. Candidate, evidence, validation, execution, errors, and latency are persisted in `AiTask.result_data`.

The model never receives database access and cannot mutate payments. Mercado Pago webhooks and explicit professional actions remain the only payment writers.

## Known migration boundary

Proactive tasks still use `Ai::ContextBuilder` and the legacy decision contract. They should be migrated one trigger at a time after the grounded inbound path has production evidence. Confirmation tasks are the recommended next stage, followed by reminders, no-response follow-ups, rescheduling, payment reminders, and post-session feedback.
